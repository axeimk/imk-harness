# ADR-0017: hooks 作成支援を 3 本目の共有スキルとして追加する

- ステータス: 承認済み
- 日付: 2026-07-21

## コンテキスト

harness-check の手順 10 は permissions / hooks の提案を含むが、hooks について
書いてあったのは各ツールの設定ファイルの場所だけで、スキーマ・イベント名・
入出力プロトコルの参照先が無かった。実際に採用プロジェクトで harness-check を
実行した際、エージェントが「Codex の hooks ルールがわからない」として
Claude Code 分の hooks しか作成しない失敗が起きた。

原因は構造的で、エージェント個体の問題ではない:

- Codex / Cursor の hooks 仕様は新しく流動的で、モデルの学習知識では書けない
  （または古い知識で幻覚する）
- 3 ツールの hooks は思想が似ている（stdin JSON / exit 2 でブロック）のに、
  設定構造・イベント名・出力スキーマがすべて異なる「似て非なる」仕様で、
  1 ツール分の知識からの類推が誤りを生む（調査記録: `docs/hooks-spec/`）
- 参照資料が実行時に手元に無い限り、どのエージェントでも再発する

これは imk-skill-creator が解決した問題（3 ツールで書式が違う成果物を、
参照資料 + 手順で正しく作らせる）と完全に同型である。

## 決定

**ユーザースコープに展開する共有スキルに imk-hooks-creator を追加し、3 本とする。**
ADR-0016 の「スキル 2 本のみ」は本 ADR により「3 本」に改訂する（ミッション
「プロジェクトスコープのハーネス育成」は変えない — hooks はプロジェクトスコープの
構成要素であり、ツール間差異の吸収はまさに汎用層の役割）。

1. **imk-hooks-creator スキル** — imk-skill-creator と同型の構成。
   `references/` に 3 ツールの hooks 仕様ダイジェスト（調査日と一次情報 URL 付き）を持ち、
   本文は「書く前に対象ツールの references を必ず読む」を鉄則とする手順を定める
2. **harness-check 手順 10 は imk-hooks-creator へ委譲する**（「スキルが無ければ
   公式ドキュメントで確認してから書く」のフォールバック付き — validate-skill.sh の
   委譲と同じ「あれば」パターン）
3. **仕様ダイジェストの正はスキルの references が持つ**。`docs/hooks-spec/` は
   調査の記録（出典・要点）だけを残し、仕様の複製を置かない（docs/skills-spec/ と
   違い、実行時に読まれる消費者がスキル側にしかいないため。複製は乖離するだけ）
4. **仕様変化への対処もスキルに埋め込む** — 記載どおりに動かないときは一次情報を
   確認し、references 側を更新する指示を本文に持つ

## 検討した代替案

- **harness-check の references に hooks 仕様を埋める** — hooks の作成・修正は
  ブートストラップ時よりプロジェクト運用中（「lint を hooks で強制して」）に多く発生する。
  harness-check は明示起動専用（description で自発起動を禁止）なので、運用中の依頼で
  発見されず、再発を防げない
- **docs/hooks-spec/ に完全な仕様を置き、スキルは持たない** — `docs/` は採用
  プロジェクトに展開されず、実行時のエージェントから見えない。今回の失敗を防げない
- **imk-skill-creator に hooks の守備範囲も足す** — 成果物（スキル / hooks 設定)も
  発火語彙（「スキル化して」/「自動で強制して」）も別物で、description の発火精度を
  双方悪化させる

## 結果

- ハーネスの実体は「スキル 3 本 + ユーザー常駐指示の配布装置 + settings の初期配置」になる
- トレードオフ: references の仕様ダイジェストは陳腐化する。緩和として各ファイルに
  調査日・一次情報 URL を記載し、食い違い発見時に references を更新する運用を
  スキル本文に定めた（それでも能動的な追従は将来課題）
- ADR-0016 のステータスに本 ADR による改訂（2 本 → 3 本）を注記する
