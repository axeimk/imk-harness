# ADR-0012: 自動テストを bats で導入する（npm devDependency 管理）

- ステータス: 承認済み
- 日付: 2026-07-17

## コンテキスト

これまで自動テストはなく、正しさの担保は検査スクリプト（`check-shell.sh` /
`check-skills.sh`）と、verify スキルの fake HOME レシピを毎回手で実行する
動作確認だけだった。verify スキルの「確認すべき代表フロー」（dry-run で FS 不変、
冪等性、ブロック外テキスト保持、ツール縮小時の掃除、ユーザーの持ち物の不可侵）は
実質的にテスト仕様であり、リグレッション検知を人手に頼る状態だった。

また、CLI 出力の整形などライブラリ全体に手を入れるリファクタリングを安全に
行うためにも、挙動を固定する回帰テストが必要になった。

## 決定

**bats-core を npm の devDependency として導入し、`tests/*.bats` に回帰テストを置く。**

- 実行は `./test.sh`（初回のみ `npm install` が必要）。`npm test` でも同じ
- 各テストは `mktemp -d` の一時ディレクトリを `$HOME` に差し替えて実行する。
  実ホームには触れず、テスト間の状態共有もない（fake-home.sh の状態再利用とは別物）
- アサーションは原則ファイルシステムの結果（symlink 先・ブロックの中身・
  `find | sort` のスナップショット比較）に対して行い、出力文字列への依存は
  「変更はありません」等の契約的なメッセージに限る。出力の見た目の変更で
  テストが壊れないようにするため
- **テスト名（`@test "..."`）は ASCII にする。** bats はテスト名をエンコードして
  関数名にするが、macOS 標準 bash 3.2 は多バイトのテスト名を解決できず
  「unknown test name」で全テストがスキップされる（ロケール設定では回避不能。
  本リポジトリ既知の bash 3.2 多バイト問題と同根）。コメントや本文中の
  日本語文字列は問題ない
- verify スキルは廃止しない。役割は ADR-0007 のまま「実際に展開して観察する
  探索的な動作確認のレシピ」であり、回帰テストとは補完関係とする

## 検討した代替案

- **brew でグローバルインストール** — マシン共通の暗黙依存になる。
  プロジェクト内で完結する依存管理の方が望ましい
- **git submodule vendoring**（bats 公式推奨） — 依存は git のみで済むが、
  clone 後の `git submodule update --init` と手動のバージョン更新が必要。
  npm の方が導入・更新の作法が馴染むため npm を選択（Node.js が前提になる点は
  許容。ハーネス本体のスクリプトは引き続き Node に依存しない）
- **自前テストランナー**（check-*.sh の延長） — 依存ゼロだが、assert・
  失敗診断・テスト列挙を自作することになり、本体より育つ

## 結果

- `tests/` に helpers.bash と install / uninstall / managed-block / lib の 4 スイート
- `package.json` / `package-lock.json` を追加、`node_modules/` は gitignore
- `check-shell.sh` は `node_modules/` を検査対象から除外
- CLAUDE.md の「コマンド」「テスト」節と verify スキルの記述を更新
