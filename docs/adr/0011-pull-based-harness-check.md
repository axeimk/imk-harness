# ADR-0011: 特化層整備を harness-check に一任し、採否を HARNESS.md に記録する

- ステータス: 承認済み
- 日付: 2026-07-17

## コンテキスト

ADR-0008 は、常駐指示（汎用層）の proactive な提案が生む再提案問題を
「提案 → 記録 → 尊重」プロトコルで管理し、採否をプロジェクト CLAUDE.md / AGENTS.md の
「ハーネス」節に記録すると定めた。

ハーネス内部の依存関係を調査した結果、次の構造的な問題が見えた。

- プロジェクトの指示ファイル（CLAUDE.md）に、プロジェクト開発の知識ではない
  ハーネス運用のメタ記録が混ざる。「ファイル内の特定の節」への参照は、
  節名の変更やユーザーの自由編集で壊れやすい
- CLAUDE.md を作らない選択をしたプロジェクトでは記録先が無く、再提案の抑止が
  セッション内に限られていた（ADR-0008 が既知の制約として許容していた点）
- 提案のトリガーが常駐指示 2 ファイル（`10-workflow.md` の verify 提案、
  `20-project-harness.md` のブートストラップ提案）と harness-check スキルに分散し、
  常駐層 → スキルの名前参照が複数残っていた
- そもそも記録・尊重の仕組みは「常駐層が proactive に提案する」こと自体への
  対症療法であり、提案の担い手を常駐層から外せば常駐層に抑止機構は要らない

## 決定

**特化層整備の提案・作成・採否記録は harness-check スキルに一任し、
ユーザーが明示的に起動する pull 型とする。採否はプロジェクトルートの
`HARNESS.md` に記録する。**

- 常駐指示から提案行動を撤去する。`20-project-harness.md` は
  「整備は harness-check が担う（ユーザーが明示起動）」という案内のみ残し、
  `10-workflow.md` の verify 作成提案は削除する
- harness-check の description を明示起動前提に書き換える
  （description 自体が各ツールで proactive トリガーとして働くため、
  「エージェントの判断で自発的に起動・提案しない」を明記する）
- 「提案 → 記録 → 尊重」は、提案を行うスキル（harness-check / domain-modeling）の
  内部の規律として存続する。記録先は `HARNESS.md`
  （可視のルート配置。CONTEXT.md と同じ流儀。雛形は `templates/HARNESS.md.template`）
- 旧「ハーネス」節が残るプロジェクトでは harness-check が移行
  （記録の移動と節の削除）を提案する。移行するまでは節の記録も尊重する
- 尊重ルール（「使わない」記録の参照）を常駐層に 1 行残す案も検討したが、
  ハーネス由来でない一般的な提案（CLAUDE.md 作成の申し出など）は許容する方針のため
  残さない（2026-07-17 ユーザー判断）

domain-modeling の proactive 動作（用語ゆらぎの指摘、CONTEXT.md 提案)は
会話の文脈起点で性質が異なるため、ADR-0010 のとおり維持する。
CONTEXT.md 提案を断られた場合はスキル内で `HARNESS.md`（あれば）に記録する。

## 検討した代替案

- **常駐層に提案を残し、記録先だけ `HARNESS.md` に移す** — CLAUDE.md との結合は
  解消するが、トリガー分散と常駐層 → スキル依存が残る。常駐層に
  「提案の前に `HARNESS.md` を確認する」という読み込み指示も必要になる
- **尊重ルールのみ常駐層に残す** — ハーネス外由来の提案にも効くが、
  ユーザーは CLAUDE.md 程度はどのプロジェクトにもあるべきと考えており、
  ハーネス由来でない提案は許容するため不要
- **旧「ハーネス」節と `HARNESS.md` の両方を参照し続ける** — 既存プロジェクトが
  無修正で動くが参照が二重になる。移行は harness-check の提案で足りる
- **隠しファイル（`.harness.md`）や `docs/` 配下** — ユーザーが見つけて編集する
  宣言的な記録なので、可視のルート配置を選んだ

## 結果

- `20-project-harness.md` から「提案 → 記録 → 尊重」節を削除し案内 1 行に縮約、
  `10-workflow.md` の verify 作成提案を削除。常駐層から `HARNESS.md` への参照はゼロ
- harness-check: description を明示起動前提に変更、現状確認と採否記録を
  `HARNESS.md` ベースに変更、`templates/HARNESS.md.template` を追加、
  旧「ハーネス」節の移行提案を追記。`CLAUDE.md.template` から「ハーネス」節を削除
- domain-modeling: CONTEXT.md 提案の抑止・拒否記録の参照先を `HARNESS.md` に変更
- ADR-0008 を「廃止（本 ADR により置換）」に変更。CLAUDE.md を作らない
  プロジェクトでも記録先を持てるようになり、ADR-0008 の既知の制約も解消
- トレードオフ: 新規プロジェクトが自動ではブートストラップされない。
  ユーザーが新規プロジェクト開始時に harness-check を明示起動する運用を選択した
