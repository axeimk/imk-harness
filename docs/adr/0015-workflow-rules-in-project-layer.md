# ADR-0015: 作業の進め方は特化層が定める（汎用層は常駐の作業ルールを持たない）

- ステータス: 承認済み
- 日付: 2026-07-19

## コンテキスト

`shared/instructions/10-workflow.md` は「作業の進め方」（テスト・lint の実行、verify
スキルによる動作確認、省略基準、検証結果の報告、破壊的操作の確認）を汎用層の常駐指示として
全プロジェクトに強制していた。

しかしこれらの運用ルールは、プロジェクトごとに最適解が変わる内容だった。

- 検証の手段（テストの有無、verify スキルの採否、hooks による自動実行）はプロジェクトの
  整備状況で異なり、汎用層の文言はその差分を吸収するための但し書き
  （「verify スキルが無ければ既知の方法で」「hooks で自動実行される場合は」等）で
  肥大化し続けていた
- 省略してよい変更の基準（ドキュメントのみ等）もプロジェクトの性質に依存する
- ADR-0011 で特化層整備の提案を常駐層から harness-check へ移した後も、
  作業ルールの強制だけが汎用層に残っており、「汎用層は特化層を育てる規約に徹する」
  （ADR-0001）という役割分担と不整合だった

## 決定

**`10-workflow.md` を削除し、作業の進め方はプロジェクト特化層（CLAUDE.md / AGENTS.md の
「作業の進め方」節）が定める。汎用層はそのデフォルト案を harness-check のテンプレートとして
提供する。**

- `20-project-harness.md` に「作業の進め方は汎用層では定めない。プロジェクト
  CLAUDE.md / AGENTS.md の『作業の進め方』節に従う」という規約を追加する
- 旧 10-workflow.md の内容は `templates/CLAUDE.md.template` の「作業の進め方」節に
  デフォルト案として移し、harness-check がプロジェクトの実態（verify の採否、hooks の有無）に
  合わせて調整して展開する。既存プロジェクトに節が無ければ追記を提案する
- スキル化の提案（同種作業の繰り返しに気づいたら）は「特化層を育てる」行動そのものなので、
  昇格ルールと並べて `20-project-harness.md` に残す

## 検討した代替案

- **汎用層にデフォルトを残し、特化層で上書きさせる** — 常駐指示が残り続け、特化層と
  二重管理になる。どちらが優先かの解釈もツール任せになる
- **正直な報告・破壊的操作の確認だけ汎用層に残す** — プロジェクトで変わらない行動規範では
  あるが、「汎用層に強制力のあるルールを置かない」という役割の線引きを崩す。
  デフォルト案に含めれば、節を導入したプロジェクトでは同じ効果が得られる

## 結果

- 汎用層の常駐指示は `00-style.md`（応答スタイル）と `20-project-harness.md`
  （特化層の育て方）だけになり、ADR-0001 の役割分担が徹底された
- 既存プロジェクトには「作業の進め方」節が無いため、harness-check の再実行で
  追記を提案する運用になる（ADR-0011 の pull 型と同じ）
- トレードオフ: 節を導入していないプロジェクトでは検証・報告のルールが効かなくなる。
  特化層の整備（harness-check の起動）がこれまで以上に前提になる
