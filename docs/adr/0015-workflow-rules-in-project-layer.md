# ADR-0015: 作業の進め方はプロジェクトスコープが定める（ユーザースコープは常駐の作業ルールを持たない）

- ステータス: 承認済み
- 日付: 2026-07-19

※ 本 ADR から用語を「ユーザースコープ / プロジェクトスコープ」に統一した（旧称:
汎用層 / 特化層。経緯は `CONTEXT.md` の「解決済みの曖昧さ」参照。過去の ADR は旧称のまま）。

## コンテキスト

`shared/instructions/10-workflow.md` は「作業の進め方」（テスト・lint の実行、verify
スキルによる動作確認、省略基準、検証結果の報告、破壊的操作の確認）をユーザースコープの
常駐指示として全プロジェクトに強制していた。

しかしこれらの運用ルールは、プロジェクトごとに最適解が変わる内容だった。

- 検証の手段（テストの有無、verify スキルの採否、hooks による自動実行）はプロジェクトの
  整備状況で異なり、ユーザースコープの文言はその差分を吸収するための但し書き
  （「verify スキルが無ければ既知の方法で」「hooks で自動実行される場合は」等）で
  肥大化し続けていた
- 省略してよい変更の基準（ドキュメントのみ等）もプロジェクトの性質に依存する
- ADR-0011 でプロジェクトスコープ整備の提案を常駐指示から harness-check へ移した後も、
  作業ルールの強制だけがユーザースコープに残っており、「ユーザースコープは
  プロジェクトスコープを育てる規約に徹する」（ADR-0001）という役割分担と不整合だった

また、常駐指示の昇格提案（「2〜3 プロジェクトで同じ指示・スキルを書いたと気づいたら
昇格を提案する」）は、エージェントがプロジェクト横断の繰り返しを観測できない
（セッションもメモリもプロジェクト単位）ため、発火し得ないトリガーだった。

## 決定

**`10-workflow.md` を削除し、作業の進め方はプロジェクトスコープ（CLAUDE.md / AGENTS.md の
「作業の進め方」節）が定める。ユーザースコープはそのデフォルト案を harness-check の
テンプレートとして提供する。**

- `20-project-harness.md` に「作業の進め方はユーザースコープでは定めない。プロジェクト
  CLAUDE.md / AGENTS.md の『作業の進め方』節に従う」という規約を追加する
- 旧 10-workflow.md の内容は `templates/CLAUDE.md.template` の「作業の進め方」節に
  デフォルト案として移し、harness-check がプロジェクトの実態（verify の採否、hooks の有無）に
  合わせて調整して展開する。既存プロジェクトに節が無ければ追記を提案する
- スキル化の提案（同種作業の繰り返しに気づいたら）はセッション内で観測可能で、
  「プロジェクトスコープを育てる」行動そのものなので `20-project-harness.md` に残す
- 昇格の提案は削除する（観測できないトリガーは書いても効かない）。README にも運用ルール
  として残さない — ユーザースコープは「プロジェクトスコープの育て方」の規約だけを持つと
  宣言しており、それ以外のルールを持ち込まない。何をこのリポジトリへ昇格させるかは
  ユーザーの判断に委ねる（経緯としての昇格ルールは ADR-0001 に記録が残る）

## 検討した代替案

- **ユーザースコープにデフォルトを残し、プロジェクトスコープで上書きさせる** —
  常駐指示が残り続け、プロジェクトスコープと二重管理になる。どちらが優先かの解釈も
  ツール任せになる
- **正直な報告・破壊的操作の確認だけユーザースコープに残す** — プロジェクトで変わらない
  行動規範ではあるが、「ユーザースコープに強制力のあるルールを置かない」という役割の
  線引きを崩す。デフォルト案に含めれば、節を導入したプロジェクトでは同じ効果が得られる

## 結果

- ユーザースコープの常駐指示は `00-style.md`（応答スタイル）と `20-project-harness.md`
  （プロジェクトスコープの育て方）だけになり、ADR-0001 の役割分担が徹底された
- 既存プロジェクトには「作業の進め方」節が無いため、harness-check の再実行で
  追記を提案する運用になる（ADR-0011 の pull 型と同じ）
- トレードオフ: 節を導入していないプロジェクトでは検証・報告のルールが効かなくなる。
  プロジェクトスコープの整備（harness-check の起動）がこれまで以上に前提になる
