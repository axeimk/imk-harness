# ADR-0014: TDD 手順を特化層の tdd スキルとして提供する

- ステータス: 承認済み
- 日付: 2026-07-18

## コンテキスト

TDD は多くのプロジェクトで有用な実装手法だが、採否はプロジェクトごとに分かれる
（プロトタイプ、テスト基盤の無いリポジトリなど、馴染まない場面がある）。また
実際にサイクルを回すのに必要な知識はプロジェクト固有のもの（単一テストの実行
コマンド、テストファイルの置き場所・命名、モック等の流儀）で、汎用層にはこれを
置く場所がなかった。

## 決定

**harness-check のチェックリストに tdd スキルを加え、採用したプロジェクトでは
`templates/tdd-SKILL.md.template` から特化層スキルとして作成する。**

- スキル本体はプロジェクト固有レシピ — Red → Green → Refactor をそのプロジェクトの
  コマンド・規約で回すための手順。配置・symlink・作成後検査の規約は verify スキルと同じ
- 常駐指示（プロジェクト CLAUDE.md / AGENTS.md）にはトリガー 1〜2 行だけを書き、
  サイクルの詳細はスキル側が持つ（ADR-0013 と同じ分担）
- 拒否は `HARNESS.md` に記録し、以後再提案しない（ADR-0011 のプロトコル）

## 検討した代替案

- **汎用層に単一の tdd スキルを置く** — TDD の一般論はモデルが既に知っており、
  価値は「このプロジェクトでどう回すか」の固有部分にある。また全プロジェクトに
  常駐するため、TDD を使わないと決めたプロジェクトの記録との整合が取りにくい
- **常駐指示に「TDD で進める」とだけ書く** — 単一テストの回し方・置き場所などの
  実務知識の置き場が無く、常駐層に書くと肥大する（ADR-0013 で退けた構図と同じ）
- **hooks で強制する** — TDD は編集後の機械検査ではなく実装の進め方であり、
  hooks のモデルに載らない

## 結果

- harness-check: 現状確認に tdd スキルを追加、作成手順（手順 6）と
  `templates/tdd-SKILL.md.template` を追加、`HARNESS.md.template` に記録行を追加
- テスト基盤がまだ無いプロジェクトでは、先にテストコマンドの整備を提案してから
  導入する（テンプレートが前提とする「単一テストの実行」が成立しないため）
