# ADR-0005: install / uninstall はプラン表示 → 確認 → 適用の 2 段階実行

- ステータス: 承認済み
- 日付: 2026-07-12

## コンテキスト

install.sh / uninstall.sh はホームディレクトリの設定ファイル（CLAUDE.md、settings.json、
スキルの symlink 等）を書き換える。当初は実行しながらログを流すだけだったが、
「ファイルに変更が生じる場合は実行前にすべて表示して確認したい」という要求があった。
また、手動対応が必要な案内（設定のマージ等）がログに埋もれる問題もあった。

## 決定

**同一の `apply_changes()` を dry-run で 1 回、承認後に本番で 1 回実行する 2 段階方式にする。**

1. `DRYRUN=1` で `apply_changes` を実行 — 変更系ヘルパーはファイルに一切触れず
   `PLAN` 配列に予定を積む
2. プランを一覧表示。**変更ゼロなら「変更はありません」で即終了**（確認も不要 = 冪等）
3. `y/N` 確認（`--yes` でスキップ、`--dry-run` は表示のみで終了）。N なら無変更で中止
4. `DRYRUN=0` で同じ `apply_changes` を本実行し、手動対応が必要な項目（NOTICES）を
   最後にまとめて表示する

変更系ヘルパー（link / copy_if_absent / write_managed_block / prune 等）はすべて
`lib.sh` に集約し、各自が dry-run 分岐を持つ。dry-run はファイルシステムを変更しないため、
プランと本実行の判定は必ず一致する。

## 検討した代替案

- **操作ごとに逐次確認** — 確認回数が多く煩雑。全体像も見えない
- **確認なし＋バックアップのみ** — 復旧は可能だが、何が起きるかを事前に把握できない

## 結果

- 実行前に全変更が見え、意図しない変更を承認前に止められる
- 再実行が冪等になり、「変更なし」が明示される（アップデート運用が安全）
- 変更系操作を lib.sh のヘルパー経由に限定する規律が必要（直接 `rm` / `cp` を書かない）
