# ADR-0016: ミッションを「プロジェクトスコープのハーネス育成」に特化する

- ステータス: 承認済み（展開スキルの本数は ADR-0017 により 2 本 → 3 本に改訂）
- 日付: 2026-07-19

## コンテキスト

このリポジトリの本質は「各プロジェクトのプロジェクトスコープを育てるためのハーネス」だと
README で宣言していた（ADR-0001）。しかし実態は、そのミッションに直接寄与しない資産を
複数抱えていた。

- 展開スキル 5 本のうち、ミッションの装置と言えるのは harness-check と
  imk-skill-creator の 2 本だけ。domain-modeling / grilling は汎用の作業技法、
  hunk-review は特定ツール（hunkdiff）の操作ラッパーだった
- 常駐指示 `20-project-harness.md` は「プロジェクトスコープの育て方」の規約 7 項目を
  全セッションに常駐させていたが、大半は harness-check の手順とテンプレートという
  **構造**に既に埋め込まれた内容の重複だった

ADR-0011（pull 型化）・ADR-0015（作業ルールの特化層移管）で進めてきた
「常駐を減らし、整備は明示起動の harness-check に集約する」方向を完成させる。

## 決定

**ユーザースコープに展開するスキルは harness-check と imk-skill-creator の 2 本のみとし、
ハーネス提供の常駐指示はゼロにする。**

1. **domain-modeling / grilling は harness-check のテンプレートに移す**
   （`templates/domain-modeling/`・`templates/grilling/`）。verify / tdd と同列の
   「harness-check が提案し、採用したプロジェクトに複製して育てる」対象になる。
   採否は HARNESS.md に記録する。中身は汎用のデフォルト案だが、プロジェクトの流儀に
   合わせた調整・育成を前提とする
2. **hunk-review はリポジトリから削除する**。「Hunk を使うか」はプロジェクトの属性では
   なくユーザー個人のレビューワークフローなので、プロジェクトスコープ行きも誤り。
   必要ならホーム側の自由編集エリアで個人管理する
3. **`20-project-harness.md` を削除し、常駐指示の残留分もゼロにする**。各行の行き先:
   - スコープ規律（固有知識はプロジェクトへ）— エージェントの自然な動作に任せ、規約と
     しては本リポジトリの README / ADR にのみ残す
   - harness-check の案内 — スキルの description（常駐メタデータ）が既に担う
   - 「作業の進め方」節への委譲 — エージェントはプロジェクト CLAUDE.md に元々従う
   - CONTEXT.md の語彙ルール — CONTEXT.md を採用したプロジェクトの CLAUDE.md /
     AGENTS.md に harness-check が 1 行書き込む
   - スキル化提案（ADR-0015 で常駐に残した行）— `templates/CLAUDE.md.template` の
     「作業の進め方」節へ移す
4. **指示配布の仕組み（`shared/instructions/` → build.sh → 管理ブロック）は残す**。
   位置づけを「ハーネスのデフォルト指示の配布」から「**ユーザー自身の常駐指示を
   3 ツールへ一元配布する装置**」に再定義する。`00-style.md`（応答スタイル）は
   ハーネス提供物ではなくユーザーコンテンツ（記入例）として存続する

## 検討した代替案

- **指示配布の仕組みごと撤去する** — 常駐指示ゼロなら ADR-0002 / 0004 の機構は空回り、
  という案。ユーザーが自分の常駐指示を 1 箇所で書き 3 ツールに配れる価値が残るため、
  中身ゼロで仕組みだけ残す方を選んだ
- **hunk-review もテンプレート化して harness-check の提案対象にする** — 導入判断が
  プロジェクトの属性でないため誤ったスコープになる
- **常駐指示に規約 3 行（スコープ規律・harness-check 案内・作業の進め方の委譲）を残す** —
  3 行とも既に構造・仕様（description、ツールの標準動作）で担保されており重複
- **domain-modeling / grilling を単一原本から無調整で複製する** — 乖離は防げるが、
  プロジェクトごとに育てる verify / tdd と扱いが割れる。テンプレート（調整前提）に統一した

## 結果

- ハーネスの実体は「スキル 2 本 + ユーザー常駐指示の配布装置 + settings の初期配置」という
  ミッションをそのまま体現する最小形になった
- ADR-0010 は配置の決定（ユーザースコープ展開・domain-modeling の常駐自動発火）が
  本 ADR で置換される。輸入した規律・逸脱の判断はテンプレートの中身として存続する
- ADR-0011 の「domain-modeling の proactive 動作は維持する」は、導入したプロジェクトの
  スコープ内でのみ効く形に変わる
- トレードオフ: domain-modeling / grilling は導入プロジェクトでしか使えず、複製後の
  改善は自動では伝播しない（harness-check の再実行時に差分を案内する運用は将来課題）
