# ADR-0007: プロジェクト verify スキルは動作確認レシピとする

- ステータス: 承認済み
- 日付: 2026-07-12

## コンテキスト

当初、harness-check がブートストラップするプロジェクト verify スキルは「テスト・lint・
ビルドコマンドを順に実行する手順書」だった。しかし Claude Code にはビルトインの verify
スキルが存在し、次の設計になっている:

- 検証 = 実行時観察（アプリを起動し、変更コードが通る経路を操作して観察する）。
  テストや typecheck の再実行は CI の再現であり検証とみなさない
- プロジェクトの `.claude/skills/verify/` を「ビルド・起動・操作のレシピ」として参照し、
  無ければ自らブートストラップする

つまり同じ名前・場所（`.claude/skills/verify`）に対して、ハーネス側テンプレートが
ビルトインと逆方向の内容（テストコマンドの羅列）を書き込む衝突があった。
また、テスト・lint コマンドはプロジェクト CLAUDE.md の「コマンド」節にも書く設計で、
内容が重複していた。

## 決定

**プロジェクト verify スキルの役割を、ビルトイン verify の期待に揃える。**

- verify スキル = そのプロジェクトを実際にビルド・起動・操作して変更を確認するレシピ
  （起動コマンド、確認すべき代表フロー、落とし穴）。テストコマンドの羅列にしない
- テスト・lint・ビルドのコマンド一覧はプロジェクト CLAUDE.md / AGENTS.md の
  「コマンド」節に置く（従来どおり。verify スキルには重複させない）
- 機械的に強制したい検査（編集後の lint など）は各ツールの hooks に設定する
  （Claude Code: `.claude/settings.json`、Codex: `hooks.json` / `config.toml` の
  `[hooks]`、Cursor: `.cursor/hooks.json`）。テストスイート全実行のような重い処理は
  hooks に入れない。harness-check では任意の提案項目とする

これにより、Claude Code ではビルトイン verify がプロジェクト verify スキルを
レシピとして参照でき、ビルトインの無い Codex / Cursor では同スキルがそのまま
動作確認の手順書として機能する。

## 検討した代替案

- **verify スキルを廃止しビルトインのブートストラップに任せる** — Codex / Cursor に
  ビルトイン verify は無く、両ツール向けの手順書が失われる
- **別名（check 等）に改名して共存させる** — ビルトイン verify はプロジェクトの
  `verify` という名前を探しに来るため、改名すると参照されず、ビルトインが別途
  verify スキルを生成して二重管理になる
- **テストコマンドと起動レシピの 2 部構成にする** — テストコマンドが CLAUDE.md と
  重複し、「テストを回すな」というビルトインの前提とも噛み合わない

## 結果

- `templates/verify-SKILL.md.template` は動作確認レシピの形式に改訂
- `10-workflow.md` は「テスト・lint は CLAUDE.md のコマンド（または hooks）」と
  「動作確認は verify スキルのレシピ」を分離して指示
- ビルトイン verify の仕様変更（参照場所や前提の変化）があれば本 ADR を見直す
