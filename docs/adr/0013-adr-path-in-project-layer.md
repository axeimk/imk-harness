# ADR-0013: ADR 記録の経路を特化層に常駐させる（harness-check で採否確認・規約 README 配置）

- ステータス: 承認済み
- 日付: 2026-07-18

## コンテキスト

ADR-0010 で ADR の提案は domain-modeling スキルが担うと決めたが、記録の経路が
このスキルの自発発火 1 本に依存していた。ハーネスを導入していない環境
（同僚、CI 上のエージェント）には経路自体が無く、導入済みでも description ベースの
自発トリガーが確実に効く保証はツールによって異なる。また CONTEXT.md と違い、
ADR 提案を断られたときの記録先が未定義で、ADR-0011 の `HARNESS.md` 方式と
非対称だった。

## 決定

**ADR の採否確認を harness-check のチェックリストに加え、採用したプロジェクトでは
記録の規律を特化層自身に常駐させる。**知識は読まれるタイミングで分担する:

- **常駐指示（プロジェクト CLAUDE.md / AGENTS.md）はトリガーだけを持つ** —
  「設計上の決定は `docs/adr/` に記録する。基準と形式は `docs/adr/README.md` に従う」
  の 2 行以内。毎セッション読まれる場所には発火条件だけあればよい
- **基準・形式・運用は `docs/adr/README.md` が持つ** — harness-check が採用時に
  `templates/adr-README.md.template` から配置する。記録する 3 条件、連番・廃止
  ステータスの運用、軽量形式（タイトル + 1〜3 文）、索引表を自己完結で記載する。
  書き手は ADR を書くその場でこれを読むため、圧縮する必要がない
- 拒否されたら `HARNESS.md` に記録し、domain-modeling も同じ記録を尊重する
  （CONTEXT.md と同じプロトコル）

整理された蓄積は「規約 README + 既存 ADR という手本」で維持される。
2 本目以降のエージェントは既存ファイルの形に倣うため、コールドスタート
（1 本目）さえ README で支えれば形式は収束する。

## 検討した代替案

- **ブートストラップ時に空の `docs/adr/` を作る** — その時点で記録すべき決定が無く、
  空のまま放置されるか埋め草 ADR を招く
- **基準・形式まで CLAUDE.md の常駐節に書く** — 常駐層が肥大する。1〜2 行に
  圧縮すると今度は形式が維持できない（本 ADR の分担が両方を解く）
- **domain-modeling のみに経路を置き続ける（現状維持）** — スキル未導入環境と
  自発発火の不確実性で記録機会が失われる。本 ADR の動機
- **harness-check 採用時も「最初の ADR まで何も作らない」を貫く** — 1 本目に手本が
  無く、常駐 2 行だけでは整った蓄積にならない。README は空スキャフォールドではなく
  規約の実体なので、domain-modeling の「最初の ADR まで作らない」原則とは矛盾しない

## 結果

- harness-check: 現状確認に ADR 置き場を追加、導入手順（手順 7）と
  `templates/adr-README.md.template` を追加、`HARNESS.md.template` に記録行を追加
- domain-modeling: `HARNESS.md` の「ADR: 使わない」を尊重し、断られたら記録する
  ルールを追記。**明示的な採否確認を経ない自発提案の経路では、従来どおり最初の
  ADR が必要になるまでディレクトリを作らない**（README 配置は harness-check 経由で
  ユーザーが採用を選んだ場合だけ）
