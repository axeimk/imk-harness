# ADR-0001: 汎用層と特化層の 2 層構造

- ステータス: 承認済み
- 日付: 2026-07-12

## コンテキスト

このリポジトリは「対話的に起動した Claude Code / Codex / Cursor がタスクをうまくこなせるように
整えた設定・知識・規約の一式（＝ハーネス）」を管理する。汎用的なハーネスを 1 つ作るか、
プロジェクトごとに特化して作るかが論点だった。

前提として、各ツール自体がユーザー層（`~/.claude/` `~/.codex/`）とプロジェクト層
（リポジトリ内の `.claude/` や CLAUDE.md / AGENTS.md）の 2 層マージを前提に設計されている。

## 決定

**どちらか一方ではなく 2 層構造とし、このリポジトリは汎用層（ユーザー層）のみを持つ。**

- 汎用層に置くもの: 個人的な好み（応答スタイル等）、全プロジェクト共通の permissions、
  そして「特化層の育て方」の**規約**
- 特化層（各プロジェクトのリポジトリ）に置くもの: ビルド・テスト方法、アーキテクチャ知識、
  プロジェクト固有スキル
- 汎用層は**規約（インターフェース）を定め、特化層が実装する**。例:「どのプロジェクトにも
  verify スキルを持たせ、作業後は必ず実行する。なければまず作る」— この育て方自体を
  汎用層の指示と harness-check スキルに実装する
- 昇格ルール: 2〜3 プロジェクトで同じ指示・スキルを書いたと気づいた時点で汎用層へ移す。
  最初から汎用層を作り込まない

## 検討した代替案

- **汎用ハーネスにすべて書く** — プロジェクト固有知識（コマンド、触ってはいけない場所）は
  汎用化できず、常駐コンテキストが肥大化する
- **プロジェクトごとに完結させる** — 好みや共通ルールのコピペが増殖し、更新が行き渡らない

## 結果

- 汎用層は小さく保たれ、新規プロジェクトの立ち上げは `/harness-check` で半自動化される
- 特化層の品質は各プロジェクトでの運用に依存する（規約で誘導はするが強制はしない）
