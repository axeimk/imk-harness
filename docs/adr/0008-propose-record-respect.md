# ADR-0008: 汎用層は強制しない — 提案 → 記録 → 尊重

- ステータス: 承認済み
- 日付: 2026-07-12

## コンテキスト

汎用層の指示には proactive な振る舞いがある（verify スキルが無ければ作成を提案する、
特化層が未整備ならブートストラップを提案する、など）。文言上は「提案」でも、
採否がどこにも保存されないため、セッションやツールをまたいで毎回同じ提案が繰り返され、
ユーザーにとっては実質的な強制（ナッジの連打）になっていた。

ハーネスの規約はユーザーの作業を助けるデフォルトであって、義務であってはならない。

## 決定

**proactive な指示は「提案 → 記録 → 尊重」のプロトコルで書く。**

- **提案**: 欠けているものに気づいたら作成を一度提案する。勝手に作らない
- **記録**: 採否はプロジェクト CLAUDE.md / AGENTS.md の「ハーネス」節に記録する
  （例: `- verify スキル: 使わない（2026-07-12 ユーザー判断）`）
- **尊重**: 「使わない」と記録された項目は再提案しない。扱いを変えるのは、
  ユーザーが明示的に見直しを求めたか、ユーザー自身がこの節を書き換えたときだけ

記録先をプロジェクト CLAUDE.md にする理由:

- 全ツール（Claude Code / Codex / Cursor）が毎セッション自動で読むため、
  新しい設定ファイルや読み込み機構が不要
- 人が読める・編集できる宣言的な形式で、ユーザーがいつでも選択を変えられる
- あるツールで断った選択が他のツールでも効く

今後、汎用層に proactive な指示を追加するときは必ずこの形式に従う
（提案の文言と、断られた場合の記録先をセットで定める）。

## 検討した代替案

- **文言の緩和のみ**（「必ず」→「推奨」）— 採否が保存されず再提案が続くため、
  実質強制が解消されない
- **専用設定ファイル**（`.harness.toml` 等）— ツールが自動では読まないため
  読み込み指示が別途必要になる。CLAUDE.md の 1 節で足りる
- **インストール時のモジュール選択**（指示ファイル単位で入れる/入れないを選ぶ）—
  ユーザー層全体で一律になり、プロジェクトごとの選択に応えられない。
  build/install の統合も必要で複雑。ユーザー層ごと外したい需要が実際に出たら
  別 ADR として検討する

## 結果

- `20-project-harness.md` にプロトコル本文を記載し、`10-workflow.md` の verify 提案と
  `harness-check` スキル（項目別の要/不要選択・採否記録の手順）をこれに合わせて改訂
- `CLAUDE.md.template` に「ハーネス」節の雛形を追加
- CLAUDE.md を作らない選択をしたプロジェクトでは記録先が無く、
  再提案の抑止はセッション内に限られる（既知の制約として許容する）
