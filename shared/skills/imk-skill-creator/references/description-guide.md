# description の書き方 — 発火精度を上げる

## 原則

- **description が発火の唯一の機構。** エージェントに常駐するのは name + description だけで、
  本文は発火した後にしか読まれない。「いつ使うか」を本文に書いても発火判断には
  一切影響しない — すべて description に書く
- 「何をするか」と「いつ使うか」の両方を書く。「いつ使うか」には具体的なトリガー文脈
  （ユーザーの言い回し、状況、ファイル形式）を含める

## 押し気味（pushy）に書く

エージェントにはスキルを**使わなさすぎる**傾向がある（under-trigger）。
控えめな description は発火しない。使ってほしい状況を具体的に列挙し、
「明示的に頼まれなくても使う」ことを書く。

悪い例:

> 社内データを表示するダッシュボードを作る方法。

良い例:

> 社内データを表示するダッシュボードを作る方法。ユーザーがダッシュボード、データ可視化、
> 社内メトリクスに言及したとき、また「ダッシュボード」という言葉を使わなくても
> 何らかの社内データを表示したいときは必ずこのスキルを使う。

## トリガー検査（机上チェック）

description を書いたら、想定クエリと突き合わせて検証する:

**発火すべきクエリ（3〜5 件）** — カバレッジを考える:

- 同じ意図の異なる言い回し（フォーマル / カジュアル / 口語や typo 混じり）
- スキル名やファイル形式を明示しないが、明らかにこのスキルが必要なもの
- 実在感のある具体的なクエリにする。ファイルパス・状況説明・列名など詳細を含む、
  実際のユーザーが打ちそうな文にする（抽象的な一言「データを整形して」はテストにならない）

**発火すべきでないクエリ（3〜5 件）** — near-miss が最も価値が高い:

- キーワードは重なるが、実際には別の道具が適切なケース
- 隣接ドメインの依頼。スキルが扱う対象に触れるが文脈が違うもの
- 明白に無関係なクエリ（PDF スキルに対する「フィボナッチ関数を書いて」等)は
  何も検証しないので入れない

各クエリについて「name + description だけを見たエージェントはこのスキルを開くか」を
判定し、ズレがあれば description を直す。判定に迷うクエリが多いのは
description が曖昧なサイン。

## 知っておくべき発火の性質

エージェントは、自力で簡単にこなせるタスクではスキルを参照しないことがある。
1 ステップの単純な依頼（「このファイルを読んで」）は description がどれだけ合致していても
発火しにくい。複数ステップ・専門知識が要るタスクで初めて安定して発火する。
想定クエリはスキルの助けが実際に要る程度に複雑なものにすること。
