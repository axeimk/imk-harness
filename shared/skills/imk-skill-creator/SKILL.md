# imk-skill-creator — 3 ツール互換スキルの作成・改善

スキル（SKILL.md + 補助ファイル）を作成・改善するための手順。作成するスキルは
**同一ファイルが Claude Code / Codex / Cursor の複数ツールから読まれても
本文が成立する**ように書く（互換規約: `references/conventions.md`）。
単一ツールでしか使わない予定のスキルでも、共通部分に寄せておくと後からの共有が容易になる。

## 設計原則

- **簡潔第一**: コンテキストウィンドウは公共財。スキルはシステムプロンプト・会話履歴・
  他スキルのメタデータと文脈を分け合う。読み手のエージェントは既に賢い前提で、
  知らないことだけを書く。各段落に「この説明はトークンコストに見合うか」を問う
- **自由度の設計**: タスクの壊れやすさに合わせて指示の具体度を決める。
  複数の正解があるタスクは方針レベルの指示（高自由度）、壊れやすく順序が重要な操作は
  具体的なスクリプト（低自由度）。崖のある細い橋にはガードレールを、開けた野原には道標だけを
- **段階的読み込み**: メタデータ（name + description）は常駐、本文は発火時、
  補助ファイルは必要時のみ読まれる。SKILL.md は 500 行未満に保ち、超えそうなら
  `references/` に切り出す。切り出したファイルは SKILL.md から**いつ読むかを明記して**
  参照する（参照は SKILL.md から一段まで。100 行を超えるファイルには目次を付ける）
- **含めないもの**: README.md・CHANGELOG.md・INSTALLATION_GUIDE.md 等の補助ドキュメントは
  作らない。スキルにはエージェントが仕事をするのに必要な情報だけを置く

## 作成手順

### 1. 意図の把握

現在の会話に元になったワークフローがある場合（「今のこれをスキル化して」）は、
まず会話履歴から抽出する: 使ったツール、手順の順序、ユーザーが入れた修正、入出力の形式。
足りない部分だけを質問する。

ゼロから作る場合は、一度に全部聞かずに重要な順に質問する:

- このスキルで何ができるようになるべきか。具体的な利用例を 2〜3 件
- ユーザーがどんな発話をしたときにトリガーすべきか
- 期待する出力形式

配置先も確認する: プロジェクトスコープ（そのリポジトリの `.claude/skills/` 等）か、
ユーザースコープ（`~/.claude/skills/`・`~/.agents/skills/`、またはそれらを管理する
リポジトリがあればその原本ディレクトリ）か。複数ツールで使う場合の配置と
symlink の指針は `references/conventions.md` の「配置場所の目安」節を参照する。
なお、プロジェクトの土台整備（CLAUDE.md や verify スキルの作成）は
harness-check スキルの守備範囲なので、そちらに任せる。

### 2. 再利用資産の計画

具体例ごとに「ゼロから実行したら何をするか」を考え、毎回繰り返される部分を資産に変換する:

- 毎回同じコードを書き直す → `scripts/` に実行可能スクリプトとして置く
- 毎回同じ情報（スキーマ、API 仕様、社内規約）を再発見する → `references/` に文書として置く
- 毎回同じボイラープレート・テンプレートから始める → `assets/` に置く

### 3. 作成

- 命名: 英小文字・数字・ハイフンのみ、64 文字以内、ディレクトリ名と一致させる。
  動詞始まりの短い名前を推奨（例: `gh-address-comments`）。ツール名の前置きで明確になるなら付ける
- `templates/SKILL.md.template` を土台に SKILL.md を書く
- 本文は命令形で書く。制約には理由（why）を添える — 大文字の MUST や ALWAYS を
  乱発したくなったら、それは設計を見直すサイン。理由が伝われば読み手は応用が利く
- **互換規約に従う**: 特定ツールでしか展開されないプレースホルダや動的注入の禁止、
  Codex 固有指定の分離先など。**`references/conventions.md` を読んでから書くこと**
- description は発火精度を決める最重要フィールド。書き方は
  `references/description-guide.md` を読む
- `scripts/` に置いたスクリプトは実際に実行して動作を確認する

### 4. 機械検査

```sh
scripts/validate-skill.sh <path/to/skill-dir>
```

命名規則・description・禁止パターン・フロントマターと `agents/openai.yaml` の整合を検査する。
エラーを直して再実行し、通るまで繰り返す。

### 5. テスト

**トリガー検査（机上）**: 発火すべきクエリと発火すべきでない近接クエリを数件ずつ想定し、
description と突き合わせる。手順は `references/description-guide.md` の「トリガー検査」節。

**実地テスト**: サブエージェントが使える環境なら forward-test する。
使えない環境では、新しいセッションで代表タスクを試すようユーザーに提案する。

forward-test の作法（テストの独立性を守る）:

- サブエージェントには「スキルをテストして」ではなく、実ユーザーと同じ形で依頼する
  （例: 「<path> のスキルを使って <具体的なタスク> をして」）
- 期待する答え・疑っている不具合・自分の診断は渡さない。渡すとテストが
  「スキルが機能するか」でなく「文脈から答えを復元できるか」の測定になってしまう
- イテレーションをまたぐ成果物は掃除する（前回の出力を見つけて写されると汚染になる）
- 出力だけでなく過程も読む。スキルが無駄な行動を誘発していないかを見る

### 6. 反復

テストやユーザーのフィードバックを受けたら:

- **一般化する**: スキルは目の前の数例ではなく、無数の未知のプロンプトで使われる。
  個別例に効く小手先の修正や過度な制約を足すのでなく、失敗の原因を理解して
  一般的な指示・別のたとえ・別の作業パターンとして書き直す
- **痩せさせる**: 役に立っていない記述は削る。過程を読んで、スキルの記述が
  無駄な作業を誘発していたらその記述を疑う
- **繰り返しをスクリプトに昇格させる**: 複数のテストで似た補助コードが毎回書かれていたら、
  それを `scripts/` に一度だけ書いてスキルから使わせる

変更したら手順 4（機械検査）から繰り返す。

## 既存スキルの更新

- `name` とディレクトリ名は変えない（利用側の参照が壊れる）
- `agents/openai.yaml` がある場合、SKILL.md の変更後も記述が古くなっていないか確認する
- 更新後も手順 4〜5（機械検査・テスト）を行う
